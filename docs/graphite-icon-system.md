## 0. Overview / 系統概述

**What it is / 系統本質**
An icon system for enterprise software interfaces — clarity at small sizes, visual consistency across large inventories, and a professional register that deliberately avoids cartoon-style aesthetics.

為企業軟體介面而生的圖示系統 —— 小尺寸下依然清晰、大規模圖示庫的視覺一致性、專業穩重、明確避開卡通化美學。

**Why it exists / 為何存在**
Graphite shifts icon production from execution to rule-setting. By formalizing design judgment as rules a machine can follow, AI generates on-brand icons on demand — reducing production cost structurally, not just accelerating manual work.

Graphite 將圖示製作從「執行」轉為「定義規則」。將設計判斷形式化為機器可依循的規則,AI 就能隨需生成符合系統的圖示 —— 這不是加快人工作業,而是從結構上降低設計生產成本。

**Scope / 範圍**
- Functional UI icons only (navigation, action, status, object) / 僅適用於功能性 UI 圖示
- Illustration, brand marks, decorative graphics — out of scope / 插畫、品牌 logo、裝飾性圖形不在此系統範圍內

## 1. Design Principles / 設計原則

Graphite icons follow eight principles. Sections 1.1-1.3 define the visual register. Sections 1.4-1.8 define the taste rules that resolve edge cases.

Graphite 遵循八項原則。§1.1-1.3 定義視覺調性;§1.4-1.8 定義處理邊界情況的品味規則。

### 1.1 Enterprise register (to B) / 企業級語言

Style targets business users. Proportion and form scale to the object depicted. Avoid overly cartoon / Q-style aesthetics.

針對商業用戶。造型比例根據所描繪物件調整。避免卡通化 Q 版風格,保持專業穩重。

### 1.2 Stroke as the primary language / 線條是主要語言

Default stroke is 1.5px. Stroke weight defines the visual crispness of the entire system.

預設線寬 1.5px。線條粗細決定整個系統的視覺精緻度。

### 1.3 Rounded corners fused with square forms / 圓角融合方正

Combine rounded corners with squared elements. Proportion of roundness scales with the object's real-world softness (see §6 for grading).

圓角與方正元素結合。圓角比例依物件真實柔軟度分級調整(詳見 §6)。

### 1.4 Restraint under composition / 多元素克制

**When an icon carries 3+ elements, individual elements must be simplified** — stylized geometry over literal representation — to prevent visual noise. Cleverness (巧思) belongs to composition, not to each element.

**圖示包含 3 個以上元素時,單一元素必須簡化** —— 用幾何化的意象取代寫實表現。巧思屬於「組合」,不屬於「每一個元素」。

**Example / 範例** — `icn_photo` has frame + mountain + sun (3 elements). Mountain is stylized geometry, not literal. Sun is a single dot, no rays. Cleverness lives in how the three combine.

`icn_photo` 有相框 + 山 + 太陽三個元素。山用幾何化造型,不寫實。太陽只是一個點,不畫光芒。巧思在三者的組合。

### 1.5 Minimal expression / 極簡表達

**Complexity comes from form, not decoration.** When an element can be expressed through the outer shape's geometry alone, do not add interior detail lines.

**複雜度來自形狀本身,不來自裝飾**。當元素可以透過外輪廓幾何表達時,不加內部細節線條重複強調。

**Example / 範例** — Document icon's folded corner is expressed by the chamfered outline geometry. No interior L-shaped fold indicator line is added.

文件圖示的折角靠外輪廓的斜切幾何表達,不再畫一條內部的 L 型指示線來強調。

### 1.6 Structure by stroke, detail by fill / 結構用線條、細節用填色

**Structural elements** (outer containers, main outlines, primary shapes) are rendered as 1.5px strokes (outlined via evenodd — see §4.4).
**Interior detail elements** (dots, small shapes, indicator elements inside a container) are rendered as solid fills.

**結構性元素**(外部容器、主要輪廓、主要形狀)用 1.5px 線條(以 evenodd outlined 表達,見 §4.4)。
**內部細節元素**(點、小形狀、容器內指示元素)用實心填色。

**Rationale / 原因** — A 1.5px stroke on a small interior shape produces visual noise and disappears at reduced sizes. Fill preserves shape at any scale. This creates natural hierarchy: **outline = structure, fill = detail**.

1.5px 線條套用在小型內部形狀時,會顯得雜亂且在縮小時消失。填色在任何尺寸都能保持形狀。這自然建立了「線條 = 結構、填色 = 細節」的視覺階層。

#### 1.6.1 Exception — Aperture elements use outlined ring / 孔洞元素用 outlined ring 例外

**Trigger / 觸發**: Interior detail is a CIRCULAR APERTURE (pupil, iris, camera lens center) AND the icon appears in REPEATED UI contexts (password preview toggles, list rows, form inputs).

**觸發**:內部細節是**圓形孔洞**(眼瞳、光圈、鏡頭中心)且 icon 會**重複出現在 UI 中**(密碼欄 preview、列表 row、表單重複區塊)。

**Rule / 規則**: Use **outlined ring**, NOT solid fill.
**規則**:用 outlined ring(外圓 + 內圓 evenodd),不用實心。

**Rationale / 原因**:
- Repeated solid dots create heavy visual noise / 重複的實心點會造成視覺沉重
- Aperture metaphor is inherently a "hole" — ring communicates it better / 孔洞本質是「洞」,ring 更貼合意象
- Rings maintain visual lightness at high repetition density / 重複密度高時仍保持輕盈

**Example / 範例** — `icn_eye` for password preview: pupil drawn as outlined ring (outer r=3, inner r=1.5), NOT solid dot.

`icn_eye` 用於密碼 preview:瞳孔為 outlined ring(外 r=3, 內 r=1.5),非實心點。

### 1.7 Proportional balance / 比例平衡

**When a single small element sits below a larger main body, use a horizontal bar rather than a dot.** A single dot creates proportional imbalance against the mass above; a bar reads as intentional and balanced.

**當單一小元素位於較大主體下方時,用橫線 bar 而非點**。單一個點會與上方主體造成比例失衡;橫線讀起來刻意且平衡。

**Example / 範例** — Bell's clapper is a 4×1.5 horizontal bar, not a single dot.

`icn_bell` 的鈴錘是 4×1.5 橫線,不是單點。

### 1.8 Open bottom for body-like forms / 身體形狀的開放底部

**Vertical body shapes** (person torsos, containers standing upright, shoulder-like structures) **leave the bottom OPEN** rather than drawing a closing bottom line. Only small connector notches at corners join outer and inner walls.

**垂直的身體形狀**(人像軀幹、直立容器、肩膀狀結構)**底部保持開放**,不畫封底的橫線,僅以角落的小 notch 連接內外壁。

**Rationale / 原因** — Reduces visual clutter at the bottom edge. The shape reads as natural verticality, not as a sealed container.

減少底部邊緣的視覺雜訊。形狀讀起來為自然垂直的意象,而非密封容器。

**Example / 範例** — `icn_person` body outline has open bottom; only 1.5-wide notches at (4→5.5, 20) and (18.5→20, 20) connect inner and outer walls.

`icn_person` 身體輪廓底部開放,僅在 (4→5.5, 20) 和 (18.5→20, 20) 兩處有 1.5 寬的連接 notch。

### 1.9 Visual weight consistency / 視覺份量一致性

**Every icon's main element must maintain visual weight comparable to peer icons in the system.** When placed side-by-side, no icon should appear noticeably smaller, lighter, or emptier than its neighbors.

**每個 icon 的主要元素必須與系統內其他 icon 視覺份量相當**。放在一起檢視時,不能有任何一個明顯偏小、偏輕、或空盪。

**Rule of thumb / 判斷原則**:
- For icons whose main element is a single dominant shape (circle, rectangle, body form), scale it to fill **12-16 units** of the 16-unit live area / 主元素為單一形狀時,填滿 12-16 單位
- The system's baseline visual weight is set by the largest peer icons — **match up, not down** / 系統基準由份量最大的 icon 定義 —— **向上對齊,不向下對齊**

**Example / 範例** — `icn_magnifying_glass` lens is drawn at diameter 12 (r=6). At diameter 10 (r=5), the icon would appear lighter than folder/photo/computer/home peers.

`icn_magnifying_glass` 鏡片直徑為 12(r=6)。若使用直徑 10(r=5),放在 folder / photo / computer / home 旁邊時會明顯偏輕。

**Exception / 例外** — Subordinate elements intentionally kept small per §1.7 (e.g., bell clapper bar) are not "main elements" and do not need to meet 12-16 sizing.

依 §1.7 刻意保持較小的從屬元素(如鈴鐺橫線)不屬於「主元素」,不受此規則限制。

## 2. Anatomy / 結構

### 2.1 Canvas / 畫布
- **Size**: 24 × 24 px
- **Purpose** — Consistent bounding box across all icons. / 統一所有圖示的邊界框。

### 2.2 Padding (safe area) / 邊距
- **Value**: 4px on all sides / 四邊各 4px
- **Purpose** — Prevents icons from touching adjacent elements. / 防止圖示與相鄰元素貼合。

### 2.3 Live area / 實際繪製區
- **Size**: 16 × 16 px maximum
- **Rule** — No icon shape may extend beyond this 16 × 16 zone. / 任何圖形都不可超出這 16 × 16 的範圍。

> ⚠️ **Common mistake** — Using the full 24 × 24 canvas as drawable area causes visual weight inconsistency. / 用整個 24 × 24 畫布當繪製區,會造成系統內視覺重量不一致。

## 3. Keyline shapes / 視覺參考線

Because circles, squares, and rectangles have different perceived weights at the same pixel dimensions, they need different bounding sizes to appear visually balanced.

因為圓形、正方形、長方形在相同像素尺寸下視覺重量不同,需要不同的邊界尺寸來達到視覺平衡。

| Shape / 形狀 | Keyline size |
|---|---|
| Circle / 圓形 | 16 × 16 px |
| Square / 正方形 | 14 × 14 px |
| Rectangle (portrait) / 直長方 | 12 × 16 px |
| Rectangle (landscape) / 橫長方 | 16 × 12 px |

## 4. Stroke / 線條

### 4.1 Default weight / 預設線寬
**1.5px** — the system default.

### 4.2 Exception / 例外
**1px** — permitted only when a shape is very small or interior detail is dense and 1.5px would cause visual collapse.
僅在形狀極小或細節密集時使用。

### 4.3 Interior consistency / 內部一致性
Interior lines use the same weight as the outline. No mixed weights within a single icon.
內部線條與外框相同粗細。同一圖示內不混用不同線寬。

### 4.4 Stroke alignment (production-critical) / 筆觸對齊(生產關鍵)

**Graphite strokes are conceptually INSIDE-aligned.** Design coordinates represent the OUTER visible boundary; strokes live entirely inside that boundary.

**Graphite 線條為概念上「內對齊」**。設計座標代表外部可見邊界;線條完全落在該邊界內側。

**Why**:
- Center-aligned 1.5px stroke straddles pixel boundaries → half-pixel anti-aliasing → visible blur
- Center-align makes visible outer radius = design + 0.75 = 2.75px (not 2px)
- Inside-align guarantees pixel-perfect rendering and exact designed radius

**為什麼**:
- 中心對齊的 1.5px 線條跨過像素邊界 → 半像素抗鋸齒 → 視覺糊
- 中心對齊使實際外徑 = 設計 + 0.75 = 2.75px(而非 2px)
- 內對齊保證 pixel-perfect 渲染與設計精準的圓角半徑

**Because SVG does not natively support inside-align**, production SVGs are output as **outlined paths using evenodd fill** — an outer path + an inner path offset inward by 1.5px, filled between.

**因 SVG 原生不支援 inside-align**,產出格式為 **outlined path + evenodd 填色** —— 外路徑 + 內路徑往內縮 1.5px,以 evenodd 填色中間環狀。

**Two forms / 兩種形態**:
- **Canonical form** (design source) — 1.5px stroke SVG, used for design/editing
- **Production form** (output) — outlined evenodd SVG, used for publishing

**規範形式**(設計源)—— 1.5px stroke SVG,用於設計 / 編輯
**生產形式**(產出)—— outlined evenodd SVG,用於發佈

### 4.5.1 Path complexity safety / 路徑複雜度安全(避免破圖)

**Status / 狀態**: MANDATORY — must be checked before delivering any icon.
**狀態**:強制性 —— 每張 icon 交付前必須檢查。

**Problem / 問題**:
Certain SVG path structures cause unpredictable rendering ("broken" icons that display with wrong holes, weird fills, or missing parts). Bounding-box audit does NOT catch these — they only show visually.

某些 SVG 結構會導致渲染異常(破圖:錯誤的 hole、奇怪的 fill、缺失的部分)。**Bounding-box audit 抓不到,只有肉眼看才看得出來**。

**Common failure modes / 常見破圖情境**:

1. **Slash overlay on evenodd compound** — Trying to punch a "negation slash" (bell_off, eye_off, wifi_off) THROUGH an existing evenodd container.
   在 evenodd 容器**上**再疊「劃掉」的斜線 —— 這幾乎必破。

2. **Multiple concentric offset paths in one evenodd** — E.g., paperclip with nested U shapes as one compound path.
   單一 evenodd 內有**多層同心 offset 路徑** —— 例如 paperclip 巢狀 U 型放在一個 compound path。

3. **Ellipses (rx ≠ ry) mixed with other paths** — Elliptical arcs interact unpredictably with other elements.
   **橢圓弧(rx ≠ ry)** 與其他路徑混合 —— 互動難預料。

4. **Self-intersecting bezier** — Curves that cross themselves.
   **自交曲線** —— bezier 或折線自己交叉。

5. **Two overlapping arcs forming crescent without proper fill-rule** — Moon-type crescents drawn as 2 arcs without explicit `fill-rule="evenodd"`.
   雙弧疊成的月牙形**沒指定** `fill-rule="evenodd"` —— 預設 nonzero 會渲染錯。

**Rules / 規則**:

- **Max 2 evenodd compound paths per icon** / 每張 icon 最多 2 個 evenodd compound path
- **Max 5 total `<path>` elements per icon** / 每張 icon 最多 5 個 `<path>` 元素
- **Negation slashes are SEPARATE paths** — Never inside the same evenodd as the base shape / **劃掉斜線一律是獨立 path**,絕不與 base shape 共用 evenodd
- **Prefer circles over ellipses** / 圓比橢圓好
- **Crescent (moon-like) shapes must use `fill-rule="evenodd"` explicitly** / 月牙類形狀**必須明寫** `fill-rule="evenodd"`
- **Visual verification required** — bbox audit is necessary but NOT sufficient / **必要目視驗證** —— bbox audit 是必要條件不是充分條件

**Reference example / 反例**: `icn_bell_off` v1 was designed with bell + slash inside a single evenodd compound → rendered wrong. Fixed: bell as one outlined path, slash as separate simple path.

`icn_bell_off` 初版把鈴鐺 + 斜線放在同一個 evenodd 裡 → 破圖。修法:鈴鐺一個 outlined path,斜線獨立一個 path。

### 4.5 Sub-pixel polish (Optional / Advanced) / 次像素微調(選用 / 進階)

**Status / 狀態**: Optional — NOT applied by default in skill generation. Reserved for manual finishing pass.
**狀態**:選用 —— skill 自動生成時**不套用**,保留給人工 finishing pass。

**Problem it solves / 解決的問題**:
- When 1.5px stroke renders as outlined evenodd, outer edge at integer pixel is fully covered (100%), inner edge at half-pixel is partially covered (50%).
- Result: one side of the ring appears crisp, the other side appears soft — visually asymmetric under close inspection.

**解決的問題**:
- 1.5px stroke 用 outlined evenodd 渲染時,外邊界在整數 pixel 全覆蓋(100%),內邊界在半 pixel 部分覆蓋(50%)。
- 結果:線條一側銳利、另一側柔和 —— **近距檢視時視覺不對稱**。

**Technique / 技術**:
- Shift both outer and inner boundaries by 0.25 units perpendicular to the edge.
- Produces symmetric 25% / 100% / 25% pixel coverage — visually more uniform ring at 200%+ zoom.

**做法**:
- 將外、內邊界都往內移動 0.25 單位(垂直於邊)。
- 產生對稱的 25% / 100% / 25% pixel 覆蓋 —— 200% zoom 以上檢視時線條更均勻。

**Trade-offs / 取捨**:
- ✅ Symmetric edge rendering, professional polish at high zoom
- ✅ 邊界渲染對稱、高倍率下更精緻
- ❌ Adds sub-pixel decimals (0.25 / 0.75) throughout coordinates
- ❌ 座標充滿次像素小數
- ❌ Hard to auto-enforce — requires judgment about which edges are "stroke edges"
- ❌ 難以自動化 —— 需要判斷哪些邊是「stroke 邊」
- ❌ Imperceptible at 24px display; only visible at 48px+ or heavy zoom
- ❌ 24px 顯示下看不出來;僅在 48px+ 或大倍率下可見

**When to apply / 何時套用**:
- Manual finishing pass for portfolio-quality reference icons / 為 portfolio 級參考 icon 的人工 polish
- Icons intended for display at large sizes (48px+) / 預計大尺寸顯示的 icon
- Never auto-applied by the Graphite skill / **skill 絕不自動套用**

**Reference example / 範例**: `icn_battery` polished version has outer y-boundaries at 6.75 / 16.75 (instead of standard 7 / 17), and inner at 8.25 / 15.25 (instead of 8.5 / 15.5). Ring width remains 1.5px, but sub-pixel rendering is symmetric.

**Note / 註**: This is an industry-niche technique not widely documented in Material Icons, Feather, Lucide, or IBM Carbon. Its use signals craft-level attention to sub-pixel rendering — appropriate for pro portfolio work, over-engineered for general icon libraries.

**產業定位**:此技術在 Material Icons、Feather、Lucide、IBM Carbon 等大廠 icon 系統中皆未廣泛文件化。使用它顯示對次像素渲染的深度掌握 —— 適合 pro portfolio 作品,對一般 icon library 屬於過度工程。

Graphite generates production form directly.

Graphite 直接產出生產形式。

## 5. Stroke Logic / 線條邏輯

### 5.1 Construction order / 繪製順序

- **Solid before dashed** / 實線先於虛線
- **Left to right, top to bottom** / 由左至右、由上至下
- **Outer to inner (wrap-inward)** / 由外至內(包覆式擴展)
- **Center-align on the pixel axis** / 筆觸中心對齊像素軸

### 5.2 Why this matters / 為什麼重要

At 1.5px, pixel alignment is not automatic. If two icons use different alignment logic, one appears crisper and one appears soft — even if geometry is identical.

1.5px 下,像素對齊不會自動處理。不同對齊邏輯的兩個圖示,一個會看起來銳利、一個會看起來鬆軟 —— 即使幾何結構相同。

## 6. Border radius / 圓角(分級系統)

Border radius is **not a single default value**. It is a **graded system**: each icon's radius is chosen based on the object's real-world softness.

圓角**不是單一預設值**,是**分級系統**:每個圖示的圓角依所繪物件的真實柔軟度決定。

### 6.1 Radius grades / 圓角等級

| Radius / 半徑 | Register / 屬性 | When to use / 使用時機 | Examples / 範例 |
|---|---|---|---|
| **0px** | Hard-edged / 硬邊 | Objects that are structurally sharp in real life | House (roof, walls), screen bezel, chip, monitor edge |
| **2px** | Standard / 標準 | Default for most functional UI containers | Folder, document, mail, photo frame, computer screen |
| **4px** | Soft / 軟 | Cushioned or handleable objects | Card, wallet, bag |
| **6px** | Organic / 有機 | Body parts, dome shapes, biological forms | Person shoulders, bell dome |
| **100% / full round** | Circular / 圓 | Shape is intrinsically circular | Dot, sun, ring, avatar |

### 6.2 Rule of thumb / 判斷原則

**Ask**: "Would this object be sharp, cushioned, or organic if I touched it in real life?"

**判斷**:「如果現實中觸摸這個物件,它是尖銳、緩衝、還是有機的?」

- Sharp → 0px
- Everyday container → 2px
- Cushioned → 4px
- Body / dome → 6px
- Circular → full round

### 6.3 Inside-align implication / 內對齊的推論

Design coordinates specify OUTER radius. Inner radius = outer radius - 1.5 (stroke width).

設計座標指定外徑。內徑 = 外徑 - 1.5(線寬)。

- Outer 2px → inner 0.5px
- Outer 4px → inner 2.5px
- Outer 6px → inner 4.5px

> ❌ **Don't** use 4px radius across the whole system to feel "friendly." That drifts into consumer aesthetics.
> 為了「親切感」全系統套 4px。這會飄向消費美學。

## 7. End caps / 端點

### 7.1 Rule / 規則
- **Internal stroke end caps**: square (butt cap) / 內部線條端點:直角(平頭)
- **Outer container corners**: follow §6 border radius / 外部容器邊角:依 §6 圓角規則

### 7.2 The distinction / 兩者的區別

The outer boundary may be rounded per §6. **Internal strokes** (text lines, dividers, interior details) **always terminate with square end caps**.

外邊界可依 §6 使用圓角。**內部線條**(文字、分隔、內部細節)**永遠使用直角端點**。

## 8. Gaps / 間距

- **Safe gap**: **1.5px** minimum
- **Rule**: 1.5px minimum. May be increased for clarity, never decreased.
- **Purpose**: Preserves shape distinction at 16px display size.

安全間距 1.5px 最小值。可加大,絕不縮小。確保 16px 顯示尺寸下形狀可分辨。

## 9. Angles / 角度

### 9.1 General rule / 通則
- **Base unit**: 15°
- **Most common angle**: 45°
- **Allowed angles**: 15°, 30°, 45°, 60°, 75°, 90°(及其鏡像)

### 9.2 Arrow-specific rules / 箭頭專屬規則

Arrows in Graphite follow a specific construction pattern that differs from other icons:

Graphite 中的箭頭遵循獨立的構造模式,與其他圖示不同:

**Structure / 結構**:
- Arrow is **ONE filled path**, not separate head + shaft. / 箭頭是**單一 filled path**,不是分開的頭 + 軸
- Head and shaft are merged into a single continuous outline. / 頭與軸合為單一連續輪廓

**End caps on chevron arms / 箭頭臂端點**:
- **Square cut, perpendicular to arm direction** / **方形切斷,垂直於臂方向**
- Not slanted, not rounded / 不斜切、不圓角

**Angles / 角度**:
- Chevron arms typically at 45° / 箭頭臂通常 45°
- Shaft is vertical (90°) / 軸為垂直(90°)

**Example / 範例** — `icn_download` has tip at (12, 15.06), chevron arms at 45° with square perpendicular caps at endpoints (17.03, 10.03) ↔ (15.97, 8.97).

`icn_download` 的箭頭尖端在 (12, 15.06),箭頭臂為 45°,端點在 (17.03, 10.03) ↔ (15.97, 8.97) 的方形垂直切斷。

## 11. Naming convention / 命名規範

### 11.1 Naming philosophy / 命名哲學

Graphite naming follows the structural logic of Apple SF Symbols, adapted for enterprise UI. Names must be intuitive, extensible, and objective (describing the visual, not the function).

Graphite 命名邏輯參考 Apple SF Symbols,並針對企業 UI 情境調整。要求:直覺、可擴充、客觀描述視覺。

### 11.2 Structural pattern / 命名結構

```
[type]_[asset_size]?_[category]?_[description]_[visual_size]?_[thickness]?_[color]?
```

**Full example / 完整範例**: `icn_32_toolbar_microphone_sm_fill_red`
**Minimum example / 最簡範例**: `icn_microphone`

### 11.3 The seven fields / 七個欄位

| # | Field | Required | When to use | Examples |
|---|---|---|---|---|
| 1 | `type` | ✅ | Always | `icn`, `img` |
| 2 | `asset_size` | ❌ | Only when non-24 | `20`, `32` |
| 3 | `category` | ❌ | Scope-restricted icons | `toolbar`, `sidebar` |
| 4 | `description` | ✅ | Always | `magnifying_glass` |
| 5 | `visual_size` | ❌ | Non-standard drawing size | `sm`, `md`, `lg` |
| 6 | `thickness` | ❌ | Non-default weight or fill | `light`, `bold`, `fill` |
| 7 | `color` | ❌ | Status / colored icons | `red`, `green` |

### 11.4 Naming principles / 命名原則

1. Describe the object itself / 描述物件本體 — `icn_brush`
2. Numbers precede noun / 數字前置 — `icn_2person`
3. Mark directionality / 標註方向 — `icn_arrow_down_to_line`
4. Mark category / 標註分類 — `icn_doc_text`
5. Shape before element / 先形狀再元素 — `icn_hexagon_exclmark`
6. Badge suffix / Badge 後綴 — `icn_person_badge_plus`
7. Use proper nouns / 使用專有名詞 — `icn_namespace`

### 11.5 Status icons / 狀態圖示

```
icn_[asset_size]?_status_[description]_[color]
```

**Examples**:
- `icn_status_checkmark_green`
- `icn_20_status_exclmark_red`
- `icn_status_hourglass_grey`

**Color tokens / 色票**:
- green: `#129453`
- red: `#D13535`
- grey: `#ADB1B5`
